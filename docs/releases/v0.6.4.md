# OpenBrep v0.6.4

Release type: installation experience patch.

## Why

The previous public installation path still looked like a developer workflow:

```text
git clone → install dependencies → run obr
git pull → reinstall when dependencies change
```

That is acceptable for contributors, but it is too much friction for architects
and designers who just want to open the workbench.

## What changed

- Make GitHub Release desktop packages the recommended path for ordinary users.
- Fix installer workflow asset paths to match the actual PyInstaller zip names:
  `OpenBrep-free-macOS.zip` and `OpenBrep-free-Windows.zip`.
- On `v*` tags, automatically create or update the matching GitHub Release and
  upload installer zip assets.
- Clarify installation tiers in README and `INSTALL_CN.md`:
  desktop package for users, `pipx` / `uv tool` for command-line users, source
  install for developers.
- Bump package version to `0.6.4`.

## Recommended install paths

### Ordinary users

Download the latest desktop package from:

```text
https://github.com/byewind1/openbrep/releases/latest
```

Then unzip and run `OpenBrep`.

### Command-line users

After the package is published to PyPI:

```bash
pipx install "openbrep[ui]"
obr
```

or:

```bash
uv tool install "openbrep[ui]"
obr
```

### Developers

```bash
git clone https://github.com/byewind1/openbrep.git
cd openbrep
bash install.sh
obr
```

## Notes

- npm / npx is not the primary path for this release. OpenBrep is a Python +
  Streamlit application, so npm would only be a wrapper that adds Node.js as
  another prerequisite. It may become useful later as a bootstrapper, but it is
  not the shortest path for the current user base.
- The authoritative source remains HSF projects and Python package metadata;
  release zips are distribution artifacts.
