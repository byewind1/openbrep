# OpenBrep v0.7 Revision Management Design

Date: 2026-04-26  
Status: Draft for v0.7 implementation  
Scope: persistent project revisions, change explanation, rollback/compare command shape

## 1. Goal

v0.7 should move OpenBrep from a generation tool to a GDL asset workbench.

The minimum useful step is persistent revision management:

- Every meaningful write creates a durable project snapshot.
- Each snapshot can explain what changed and why.
- Users can compare revisions before adopting changes.
- Users can roll back to the last known good revision.
- Compile status is attached to revisions, not only shown as transient UI output.

This design intentionally stays smaller than Git integration. It should work for
non-developer users and for HSF projects that are not inside a Git repository.

## 2. Current State

The codebase already has partial revision concepts:

- `ui.app` stores `script_revision` in Streamlit session state.
- GSM compilation uses versioned file names through helpers such as
  `_versioned_gsm_path`, `_safe_compile_revision`, and `ui/project_io.py`.
- UI has an in-memory `last_project_snapshot` for one-step undo.
- Runtime modify flow already builds a readable diff summary from
  `_snapshot_scripts()` and `_build_diff_summary()`.

Limitations:

- Snapshots are not persisted as first-class project history.
- Undo is session-local and disappears after restart.
- Revision metadata is not attached to compile results.
- There is no stable CLI/API contract for `history`, `rollback`, or `compare`.
- Generated GSM versions and source HSF snapshots are not tied together.

## 3. Proposed Project Layout

For each HSF project root, create an OpenBrep metadata directory:

```text
MyObject/
  libpartdata.xml
  paramlist.xml
  scripts/
    1d.gdl
    2d.gdl
    3d.gdl
    vl.gdl
    ui.gdl
  .openbrep/
    project.toml
    revisions/
      r0001/
        manifest.json
        paramlist.xml
        scripts/
          1d.gdl
          2d.gdl
          3d.gdl
          vl.gdl
          ui.gdl
        compile.json
        explanation.md
      r0002/
        manifest.json
        ...
    latest
```

`latest` can be a small text file containing the latest revision id, for example:

```text
r0002
```

Do not use symlinks for `latest`; Windows packaging and zip export are easier
with plain text.

## 4. Revision Manifest

Each revision has a `manifest.json`.

Minimum fields:

```json
{
  "schema_version": 1,
  "revision_id": "r0002",
  "parent_revision_id": "r0001",
  "created_at": "2026-04-26T14:30:00+08:00",
  "trigger": "ai_apply",
  "intent": "MODIFY",
  "project_name": "Bookshelf",
  "instruction": "把层板改成 6 个，间距均匀分布",
  "changed_files": [
    "paramlist.xml",
    "scripts/1d.gdl",
    "scripts/3d.gdl"
  ],
  "summary": "新增 shelf_count 参数，并更新 3D 层板循环。",
  "source": {
    "ui_session_id": "",
    "model": "claude-sonnet-4-5",
    "trace_path": "traces/..."
  },
  "compile": {
    "status": "passed",
    "gsm_path": "output/Bookshelf_v2.gsm",
    "compiler": "LP_XMLConverter",
    "error_digest": ""
  }
}
```

Allowed `trigger` values:

- `project_create`
- `ai_apply`
- `manual_apply`
- `import_gsm`
- `compile_pass`
- `repair_apply`
- `rollback`

Allowed compile status:

- `not_run`
- `passed`
- `failed`
- `mock_passed`
- `mock_failed`

## 5. Snapshot Contents

Each revision stores a complete source snapshot, not only a diff.

Reasoning:

- Rollback is simple and robust.
- Users can zip/share a single revision.
- Future compare can compute diffs on demand.
- HSF projects are small enough for this to be practical.

Snapshot should include:

- `paramlist.xml`
- all non-empty `scripts/*.gdl`
- optional metadata files later: `libpartdata.xml`, `ancestry.xml`,
  `calledmacros.xml`, `libpartdocs.xml`

For v0.7 MVP, scripts and paramlist are enough. Metadata can be added after
round-trip tests confirm the exact persistence behavior.

## 6. Change Explanation

Each revision can have `explanation.md`.

Minimum template:

```markdown
# Revision r0002

## User Intent

把层板改成 6 个，间距均匀分布

## Engineering Summary

- Added `shelf_count` parameter.
- Updated Master Script to clamp shelf count.
- Updated 3D Script to draw shelves in a loop.

## Changed Files

- `paramlist.xml`
- `scripts/1d.gdl`
- `scripts/3d.gdl`

## Compile Result

Passed with LP_XMLConverter.
```

This is not a raw code diff. It is a BIM/GDL review note for users who need to
decide whether the change is safe to accept.

## 7. Runtime API Shape

Create a new module:

```text
openbrep/revisions.py
```

Suggested core API:

```python
@dataclass
class RevisionRecord:
    revision_id: str
    parent_revision_id: str | None
    project_name: str
    created_at: str
    trigger: str
    intent: str
    instruction: str
    changed_files: list[str]
    summary: str
    compile_status: str
    gsm_path: str


class RevisionStore:
    def __init__(self, project_root: Path): ...

    def next_revision_id(self) -> str: ...
    def latest_revision_id(self) -> str | None: ...
    def create_revision(
        self,
        project: HSFProject,
        trigger: str,
        intent: str,
        instruction: str,
        changed_files: list[str],
        summary: str,
        compile_result: CompileResult | None = None,
        trace_path: str | None = None,
    ) -> RevisionRecord: ...

    def list_revisions(self) -> list[RevisionRecord]: ...
    def restore_revision(self, revision_id: str, project: HSFProject) -> HSFProject: ...
    def compare(self, from_revision: str, to_revision: str) -> str: ...
```

The store should not call the LLM. It only persists and reads source state.

## 8. Pipeline Integration

Initial integration points:

### CREATE

After scripts are generated and applied:

1. Save current HSFProject.
2. Create `r0001` with trigger `project_create`.
3. Attach compile result if compile was run.

### MODIFY / DEBUG / REPAIR

Before applying changes:

1. Resolve current latest revision.
2. Build before snapshot using existing `_snapshot_scripts()`.

After applying changes and optional compile:

1. Build changed files list from returned scripts.
2. Generate engineering summary from existing `plain_text`, diff summary, and compile status.
3. Create a new revision.

### Manual Apply in UI

When user accepts `pending_diffs`:

1. Capture revision before clearing `pending_diffs`.
2. Use trigger `manual_apply` or `ai_apply`, depending on source.
3. Show revision id in the confirmation toast.

## 9. CLI Command Shape

Final command shape should be:

```bash
openbrep history ./MyObject
openbrep compare ./MyObject r0003 r0005
openbrep rollback ./MyObject r0003
```

During implementation, it is acceptable to expose the MVP under an explicit
namespace first, for example:

```bash
obrcli revision history ./MyObject
obrcli revision compare ./MyObject r0003 r0005
obrcli revision rollback ./MyObject r0003
```

The user-facing command can be promoted after the API and output format settle.

Output examples:

```text
r0005  2026-04-26 14:30  passed  ai_apply      changed: 1d.gdl, 3d.gdl
r0004  2026-04-26 13:55  failed  repair_apply  changed: 3d.gdl
r0003  2026-04-26 13:40  passed  ai_apply      changed: paramlist.xml, 3d.gdl
```

Rollback should create a new revision:

```text
r0006 rollback to r0003
```

Do not mutate history by deleting newer revisions.

## 10. UI Placement

MVP UI changes:

- Show current revision in project header: `Revision r0005`.
- In pending changes banner, show:
  - changed files
  - short summary
  - compile status if available
  - future revision id if accepted
- Add a compact history expander:
  - latest 5 revisions
  - status badge
  - rollback button

Avoid building a large revision browser in v0.7. The goal is trust, not a full
source control UI.

## 11. Compare Behavior

Use line-based unified diff for MVP.

Compare output should include:

- `paramlist.xml`
- changed `scripts/*.gdl`
- changed compile status if both revisions have compile metadata

Do not compare generated `.gsm` binaries.

## 12. Safety Rules

- Rollback always creates a new revision.
- If current project has unsaved `pending_diffs`, rollback is blocked until the
  user accepts or discards them.
- If restore fails halfway, leave current project untouched.
- Never delete revision directories from the UI in v0.7.
- Metadata write failures must not corrupt HSF project files.

## 13. MVP Implementation Order

1. Add `openbrep/revisions.py` with `RevisionStore`.
2. Add unit tests using temporary HSFProject fixtures.
3. Create revision after CREATE and MODIFY in `TaskPipeline`.
4. Add CLI `history`, `compare`, `rollback` around `RevisionStore`.
5. Add minimal UI display of latest revision id and changed files.
6. Add compile metadata attachment.

## 14. Out of Scope for v0.7

- Git integration.
- Cloud sync.
- Multi-user merge.
- Binary `.gsm` diff.
- Full revision graph.
- Partial file restore.
- Visual 3D compare.

These can come later after revisions are stable.

## 15. Acceptance Criteria

v0.7 revision MVP is useful when:

- Creating a project produces `r0001`.
- Accepting AI changes produces `r0002`.
- `history` lists both revisions with compile status.
- `compare r0001 r0002` shows changed scripts and paramlist.
- `rollback r0001` restores source files and creates `r0003`.
- Tests cover create/list/compare/rollback.
